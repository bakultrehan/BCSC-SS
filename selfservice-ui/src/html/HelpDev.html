Developers Guide

Audience

This guide is for developers and software architects who wish to develop an integration to the BC Services Card as one of the login and authentication methods for their web-application. It is expected that you would have at least a passing knowledge of token-based authentication systems such as OAuth2, OIDC or SAML. The guide elaborates on some of the available tools provided in this and other sites which help you build and test the integration to the Open ID Connect (OIDC) framework provided by the BC Services Card infrastructure

Basic operation

For all architectures the the user is redirected to, and authenticated by, the BC Services Card infrastructure independantly of your calling application.

On successful authentication, a series of token exchanges take place resulting in an access-token which can be used to get user identity information (also known as "scopes") such as email address, first and last names or mailing address. Other than the data contained in the scopes which you select, there is no data accessible from the BC Services Card infrastructure. It is generally accepted good-practice to limit the scopes requested to the minimum needed for your purposes

This self-service application enables you to register with the system and return you the necessary client ID, secret and test accounts for you to plug into your application

 

Get a Client ID and Secret

The four things you need for accessing the BC Services Card infrastructure are :

  • Client ID

  • Client Secret

  • Scopes

  • Test accounts

Development/Test

For access to the Development BC Services Card infrastructure it is a simple matter of running through the “Create new project” wizard in our self-service application. This registers your organisation and delivers the necessary items in a couple of minutes. You will need to know the following details about your own application and infrastructure to be able to complete the wizard. (Hint - you can modify your details any time, so start off simply eg. default signing, no encryption, and then ramp up complexity as needed. Perhaps even begin with one of the sample applications below ?)

  • Signing algorithm - what algorithm do you require for signing the access-token and userinfo return JWT ? We only support asymmetric encryption,the options are :
    • RS256 (default)
    • RS384
    • RS512
    • ES256
    • ES384
    • ES512
    • PS256
    • PS384
    • PS512

    If unsure, use the default RS256. Note that you will need the public key from the opposing JWKS, the URL for this can be obtained from the .well-known discovery endpoint under "jwks_uri"

  • Application URL - your home page URL
  • Redirect URI - up to 10 can be entered. Note that the BC Services Card infrastructure will not allow a redirect to a URI other than in this list for your application.
  • The user attributes you require (scopes). We describe this as "packages" - choose that which corresponds most closely to the data you need
  • The number of test accounts you will need, mostly one or 2 for initial testing.Test accounts are only relevant to the development environment and comprise an ID and a password instead of the usual BC Service card authentication mechanism which typically requires a mobile device for the second authentication factor. They are essential for testing your application without using real data.
  • Encryption and encoding algorithms. The returned data can optionally be encrypted, the only available option for encryption algorithm is RSA1_5 , and the encoding can be A256GCM (default) or A256CBC-HS512​. If using encryption you will need a JWKS set up , and the URL entered in the wizard. (there is sample code on setting up a JWKS in github repo if required)

Once you have your client ID, client secret and test accounts, you are ready to plug these together with the JWKS, access-token and userinfo endpoints into your application. These endpoints are described in the ".well-known" discovery endpoint of https://idtest.gov.bc.ca/login/.well-known/openid-configuration

Production/Live

For migrating this into production, the process involves more paperwork and rigour, and sign-off by managers, since the BC Services Card delivers personally identifiable data. But from a development perspective it is not much different, with one major difference - in production you MUST encrypt all traffic to your application, so you will need to provide a JWKS and the encryption and encoding algorithms.

Sample Applications

We have prepared a sample application and a Postman test utility which can be accessed, downloaded and run from Github .

The Postman test collection can be used for a quick check that the connections are working as expected and that the key and secret are valid. The sample javascripts application uses Express.js and Passport.js libraries and handles all aspects of the BCSC interaction including signature validation and encryption.

Each have a README.md with full instructions and are a great way to do an initial validation of your key and secret which you obtain from the BC Services card self-service application.

Testing and Troubleshooting

Testing

In order to test out your integration with BC Services Card you will need test accounts. Unlike real BC Services Card accounts (tied to real people) these do NOT need 2-factor authentication through a second device (typically mobile phone) , they are simply an ID and passcode , which allow easy testing of your integration. They are, however, only able to be used to test in your Dev environment. Once you migrate to the Production environment and are pointed at the Production BC Services Card servers, you will need to use real users for testing.

Test accounts are granted as part of the BC Services Card self-service application
